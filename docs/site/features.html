SQL EDITOR

Smart SQL editor

Built for PostgreSQL specifically — the literal masking understands dollar-quoting, nested comments, and E'' escapes, so features like the destructive-statement guard read the real query rather than raw text.

  • PostgreSQL syntax highlighting
  • Completion from the live catalog: schemas, tables, and the columns of tables in scope — including through aliases
  • Keyword and built-in function completion from a static list
  • One-key formatting
  • Multiple editor tabs with unsaved-change indicators
  • Query history and reusable snippets
  • Query parameters ($1..$n) with real bind parameters, never string interpolation
RESULT GRID

Results that stay responsive

Rows stream from a bounded store into a virtualized grid, so large or wide results don't freeze the window — and the footer counts the real total, not the cap.

  • Streaming result loading, virtualized rows
  • Column sorting, client-side, up to 50,000 rows — past that it says so rather than hanging the window
  • NULL rendered as a distinct null, never as an empty cell
  • Copy the selected cell, or the whole result as CSV or Markdown
  • Export (CSV / JSON / Markdown)
  • Query duration, row count, truncation indicators
Query performance

See why a query is slow

The plan view finds the node that actually costs you time, and names the specific problem — so you leave with a next step, not a wall of text.

Available
  • Self-time bottleneck highlighting — the busy node, whatever its type
  • Correct parallel accounting — worker time isn't double-counted as wall clock
  • Rows read then discarded by a filter
  • Disk sorts and hash spills
  • Blocks read from disk vs cache, attributed to the node that read them
  • Row estimates off by 10× or more
  • Nodes executed very many times; nodes never executed
  • JIT and trigger time — the work no plan node accounts for
  • Plan comparison between two runs, and schema diff
Planned
  • Join/sort inspection panel
  • A dedicated standalone query profiler
  • Plan history across runs of the same query

Build the EXPLAIN, don't type it

  • Every option the server takes, as a toggle: ANALYZE, BUFFERS, VERBOSE, COSTS, TIMING, WAL, SETTINGS, MEMORY, SERIALIZE, GENERIC_PLAN
  • The statement it will send is shown before you send it
  • Output as JSON, text, YAML or XML — the tree is drawn from JSON, and the panel says so plainly when you pick another
  • Save the plan as JSON, text or Markdown to attach to a ticket

It refuses combinations the server would reject

  • TIMING, WAL and SERIALIZE each require ANALYZE
  • ANALYZE and GENERIC_PLAN cannot be used together
  • Options newer than the server you're on are disabled and named — MEMORY and SERIALIZE need 17, GENERIC_PLAN 16, SETTINGS 12
  • And it warns before the dangerous one: EXPLAIN ANALYZE on an INSERT, UPDATE or DELETE really executes it. The statement is parsed with the same literal-masking the destructive-statement guard uses, so a DELETE hiding inside a comment or a string doesn't trip it and a real one doesn't slip through
Why parallel accounting matters. Under a Gather, PostgreSQL reports one “loop” per worker, and those workers run simultaneously. Multiplying by loops — which is the obvious reading — can charge a single node more milliseconds than the entire query took. TupleNest divides workers back out, so self-times add up to the execution time. Checked against live PostgreSQL 15 and 18 across eleven plan shapes, including a parallel aggregate, an external-merge disk sort, a hash spill, a nested loop repeating 20,000 times, a never-executed branch, and a trigger that consumed most of its statement's runtime. JIT timing is covered by unit tests only: the servers used report jit=on but are built without LLVM, so they never emit JIT timings.

PRODUCTION SAFETY

Safety as a workflow

Classify each connection, make production visually distinct, and let PostgreSQL itself enforce read-only when you ask for it.

  • Environment classification and a production badge
  • Read-only session characteristic (server-enforced)
  • Destructive-statement detection and confirmation dialogs
  • Transaction-state visibility and tab ownership
  • A per-connection query timeout, enforced by the server's own timer
  • Cancel a running query from the toolbar or with Esc
  • Clear warning language, and least-privilege guidance

Paste a plan

Analyse a plan without handing it to a website.

The usual way to get a query plan analysed is to paste it into someone else's server. That means sending your table names, index names, filter conditions — and sometimes literal values out of production — to a third party. TupleNest reads it locally instead.

Paste what you actually have

  • The indented text straight from psql
  • Or FORMAT JSON
  • The format is detected as you type
  • Unrecognised text says so immediately, rather than drawing a misleading tree
  • A paste that is only part of a plan is called out as one

The same analysis, no connection

  • Self-time bottleneck, disk sorts, hash spills
  • Rows read then discarded; disk-vs-cache reads
  • Row misestimates, loop counts, never-executed branches
  • JIT and trigger time — the work no node accounts for
Works against a server this app can't reach. A plan from a locked-down production box, or one a colleague sent you, analyses exactly the same as one of your own queries — no credentials, no connection, and nothing sent anywhere. Open it from the activity rail, the command palette, or the File menu; it stays available while disconnected.
How the text reader was checked — and what that check missed the first time. Eleven query shapes are captured from a live server as both text and JSON, then both are parsed and compared: node count, tree depth, loops, rows, estimates, rows removed, blocks, flags, statistics and the wording of every insight must match. Run against PostgreSQL 18 alone, that check passed. Run again against PostgreSQL 15 — plain and VERBOSE — it failed five ways, because the first set of plans happened to contain no CTE Scan at the top of a tree and no SubPlan in a target list. One of those bugs put the bottleneck badge on the wrong node. All five are fixed and each has a test that was confirmed to fail beforehand. The honest lesson is about the check, not the parser: a suite that only sees one server's output only proves things about that server.
A fragment is analysed, and labelled. Half a plan parses perfectly well, which is the danger — the analysis looks exactly as confident as one from a whole plan while its percentages are measured against the wrong total. A paste whose top was cut off, or one that stops mid-line, is analysed anyway and marked, in terms of what it does to the numbers on screen.
Import & export

Move data in and out, safely

Import

  • CSV import with header mapping
  • Inferred column types and a preview of the first rows, with the generated CREATE TABLE shown before you commit
  • Runs in a transaction and rolls back on failure
  • Planned Cancelling a running import, and streaming files larger than memory

Export

  • CSV, JSON and Markdown to a file; CSV or Markdown to the clipboard
  • Spreadsheet-safe CSV on by default (formula-injection neutralized)
  • Raw-value mode when you need it
Cells starting with = + - @ (or tab/CR) are neutralized so Excel, Sheets, and LibreOffice treat them as text, not formulas. A toggle lets you opt into raw output.
