Security-review response shipped in beta.3

Security is not a feature.
It is the foundation.

TupleNest is designed to keep credentials, queries, and database data under your control — and to be honest about where its guarantees end.

Threat model

The trust boundaries

TupleNest is a desktop client. Data crosses a handful of boundaries, and each is treated as a place where something can go wrong.

Desktop WebView

The UI. Runs under a restrictive CSP with no remote content and a small, explicit command surface into the backend.

Rust backend

Owns the PostgreSQL wire protocol, credentials, TLS, and file writes. The WebView never touches the network or the keychain directly.

PostgreSQL connection

TLS with selectable verification. In the verify modes, plaintext is refused.

SSH tunnel

Optional. Host keys are verified; a changed key is rejected.

Local disk

App database, logs, crash reports — owner-only on Unix.

Update channel

HTTPS, signed artifacts, embedded public key, anti-rollback floor.

Credential security

Passwords live in the OS keychain

  • Passwords are stored in the OS keychain (macOS Keychain, Windows Credential Manager, Linux Secret Service)
  • The app database stores only an opaque reference, never the password
  • Passwords are never written to settings files
  • Passwords are never returned to the WebView
  • A held reference is reused on Test/connect so repeated attempts don't orphan keychain entries
  • In-memory password copies are zeroized where the platform allows

Honest boundary

Once you open a database session, the credential must exist in process memory to authenticate. TupleNest minimizes and zeroizes that exposure but cannot make an in-use secret unreadable to code running as your own OS account. The keychain protects credentials at rest, not against a compromised local account.

Transport security

PostgreSQL TLS modes

Four modes, from off to fully verified. New profiles default to verify-full — they fail closed until someone deliberately relaxes them.

ModeBehaviourUse
DisabledNo TLS.Trusted local environments only
PreferAttempts TLS, may fall back to plaintext.Not recommended for remote/production
Verify CATLS required; certificate chain verified; hostname knowingly skipped.The mode SSH tunnels actually need
Verify FullTLS required; certificate and hostname verified.Recommended for production
Downgrade protection is implemented and tested. The client previously never set the SSL mode, so it defaulted to prefer and could be stripped to plaintext even under verify-full. The verify modes now require TLS and refuse a plaintext server — proven against a real ssl=off server with a negative control. The form also warns when prefer is chosen for a remote host.
SSH security

Tunnels that verify the host

  • SSH tunnels to reach databases behind a bastion
  • Known-host verification with a fail-closed policy
  • A changed host key is rejected, not silently accepted
  • No silent trust-on-first-use — the fingerprint is recorded and enforced

Key handling

Private keys are read from the path you configure and used to authenticate the tunnel. TupleNest does not enable agent forwarding. The SSH host is one of only three network destinations the app ever contacts.

Production safety model

An accident guardrail, stated honestly

  • Connections are labelled development, staging, or production
  • Read-only mode issues a PostgreSQL read-only session characteristic
  • Destructive statements are detected and require confirmation
  • The open transaction is owned by a tab; another tab can't commit it
TupleNest protects against accidental writes. For strict enforcement, connect with a database role that has no write privileges. Database permissions remain the final authority — the app is a guardrail, not a replacement for least privilege.
Local data storage

What's stored, and where

DataStorage
PasswordsOS keychain
Connection metadataLocal SQLite (owner-only on Unix)
Query historyLocal SQLite, configurable retention; secret literals scrubbed
Production SQL historyA -- tuplenest:no-history directive drops a row; a production audit log records a redacted entry
Result rowsMemory only — never written to disk
Reduced error historyLocal SQLite — title, SQLSTATE, category only; no row values
LogsLocal rotating files (owner-only on Unix); chatty transport/crypto crates capped to warn in release
Crash reportsLocal only — never uploaded
On Unix, the app-data directory is 0700 and the SQLite databases (plus WAL/SHM) are 0600, so another local user can't read your profiles, history, or logs. Windows relies on the user-profile ACL.
Privacy

Nothing leaves your machine unless you send it

What TupleNest does not do

  • No telemetry
  • No account
  • No cloud sync
  • No advertising
  • No analytics SDK
  • No remote crash upload
  • No schema upload
  • No query upload

Every network destination

  • The PostgreSQL host you configure
  • The SSH host you configure (if you use a tunnel)
  • The GitHub endpoint used to check for updates

That list is exhaustive. There is no fourth destination.

Query plans are analysed locally, including pasted ones. A plan is not innocuous: it carries table names, index names, filter conditions and sometimes literal values straight out of production. The common habit of pasting one into an analysis website sends all of that to a third party. TupleNest reads pasted plans on your machine — in psql's text format or FORMAT JSON — so a plan from a locked-down production server never has to leave it. See Paste a plan.
Desktop application security

A small, hardened surface

Restrictive CSP

object-src 'none', base-uri 'none', frame-ancestors 'none'. No remote WebView content.

No arbitrary shell

The backend exposes a small, explicit set of commands — not a general shell bridge.

Scoped filesystem

The filesystem plugin was removed; exports run through a Rust command that owns a native save dialog. The WebView has no write permission.

Escaped values

Database values are escaped on the way to the UI; CSV export neutralizes spreadsheet formula injection by default.

Native dialogs

Save/open use the OS's own dialogs, driven from the backend.

Limited command surface

Each capability is granted explicitly in the Tauri capability file — nothing is on by default.

Update security

Signed, checked, and anti-rollback

  • HTTPS update source
  • Signed artifacts with an embedded public key
  • Published SHA-256 checksums
  • Anti-rollback: refuses an update below a compiled-in floor
  • Stable releases only for auto-update
  • Manual verification options for every download
Residual, stated plainly. The version manifest itself is not yet signed, so the strongest anchor today is the integrity of the GitHub account and release. Anti-rollback and per-artifact signatures reduce the risk; a signed manifest is the remaining hardening step.

Known limitations

Where the guarantees end

Stating limits is part of being trustworthy. None of these are hidden.

  • A local user with access to your OS account may reach local metadata
  • Clipboard content is controlled by the operating system, not by TupleNest
  • Read-only mode is an accident guardrail, not a substitute for database privileges
  • Raw CSV export (when you opt out of spreadsheet-safe mode) can be interpreted as formulas by spreadsheets
  • The update version manifest is not yet signed
  • This is beta software and may contain undiscovered vulnerabilities
Responsible disclosure

Reporting a vulnerability

  • Report privately via GitHub Security Advisories
  • Please don't open a public issue for a security bug
  • You'll get an acknowledgement while it's triaged
  • Fixes are prioritized by severity and coordinated with you
  • Coordinated disclosure once a fix is available
  • The current beta line is the supported version
