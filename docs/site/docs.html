Introduction

TupleNest is a local-first PostgreSQL IDE. It runs entirely on your machine, talks directly to your database over the PostgreSQL wire protocol, and stores credentials in your operating system's keychain. There is no account, no cloud, and no telemetry.

This page is a hub. Each section is a short, practical walkthrough; where a topic is broad, it links to the relevant Features or Security section for depth.

Installation

Download the build for your platform from the Download page, then:

  • macOS — open the .dmg, drag to Applications, then right-click → Open on first launch.
  • Windows — run the installer; on SmartScreen choose “More info” → “Run anyway”.
  • Linux — chmod +x the AppImage and run it, or install the .deb.

Prefer to build it yourself? The build-from-source steps are on the Download page.

Create your first connection

Open the connection editor and fill in host, port, database, and username. Choose an environment (development, staging, or production) — production connections are shown distinctly so you always know where a query will run.

Click Test to run a staged check (DNS → TCP → authentication) before you commit. When you save, the password goes to the OS keychain and the profile stores only an opaque reference.

New profiles default to verify-full TLS — they fail closed until you deliberately relax them.

Run your first query

Type SQL in the editor and press ⌘↵ (or Ctrl+Enter). Results stream into a virtualized grid below, with execution duration and true row count. Select part of the query to run only the selection.

Explore the schema

The left explorer lazily loads schemas, tables, views, functions, indexes, and constraints. Use global search (⌘P) to jump to any object, or open the ER diagram to see foreign-key relationships as a graph.

Connection profiles

Profiles group by environment and remember TLS mode, optional custom CA, SSH settings, and the keychain reference for the password. Favourite and recent connections are one keystroke away.

TLS modes

Four modes: Disabled, Prefer, Verify CA, and Verify Full. In the verify modes, TLS is required and a plaintext server is refused. See the Security page for exactly what each mode guarantees.

SSH tunnels

To reach a database behind a bastion, enable the SSH section and provide the host, user, and private key path. Host keys are verified; a changed key is rejected rather than silently trusted.

Password storage

Passwords are stored in the macOS Keychain, Windows Credential Manager, or the Linux Secret Service — never in a config file, and never returned to the UI layer. Details on the Security page.

SQL editor

Schema-aware completion, formatting, multiple tabs, query history, snippets, and parameter binding. The editor understands PostgreSQL literals (dollar-quoting, nested comments), which is what makes the safety guard reliable.

Result grid

Streaming rows, virtualized rendering, sort/filter/search, explicit NULLs, copy cell/row/selection, and export. Large results are bounded by row, byte, and per-cell limits, with clear truncation indicators.

Query plans

Run EXPLAIN on the current statement from the command palette. Toggle ANALYZE to execute it and get real timings, and BUFFERS to see which nodes read from disk. ANALYZE defaults on for a plain SELECT and off for anything that writes — it really runs the statement.

The plan is drawn as a tree. The node marked BOTTLENECK is where time is actually spent, measured as self-time: the node's wall time minus its children's. That matters because a Nested Loop or Gather often shows a large total that belongs entirely to a child.

What the badges mean

  • BOTTLENECK — most of the execution self-time is here.
  • DISCARDED n — the node read those rows and threw them away. Usually the strongest sign an index would help.
  • DISK SORT / SPILLED — a sort or hash exceeded work_mem and went to disk.
  • HEAVY READ — blocks came from disk rather than cache, so the query is I/O bound.
  • n LOOPS — the node ran that many times; a hash or merge join may beat repeating it.
  • EST ×n OFF — the planner's row estimate missed by that factor; statistics may be stale.
  • NEVER EXECUTED — the executor short-circuited that branch.

Time outside the plan

JIT compilation and trigger time are reported separately by PostgreSQL and belong to no node, so a statement can spend most of its runtime somewhere the tree doesn't show. Both appear in the statistics panel when present.

On a parallel plan, PostgreSQL reports one “loop” per worker and those workers run concurrently. TupleNest divides them back out, so self-times sum to the execution time rather than exceeding it.

Plans can be exported as JSON (which pev2 and explain.depesz.com both read), as an indented text tree, or as Markdown.

Analysing a plan you didn't run

You can analyse a plan from anywhere — psql on another machine, a ticket, a colleague's message — without connecting to anything. Open Analyze a pasted plan from the activity rail (the clipboard icon) or the command palette, paste, and press ⌘↵.

Both formats work: the indented text psql prints, and FORMAT JSON. TupleNest detects which as you type and says so before you press anything; text it doesn't recognise is called out rather than drawn as a misleading tree. Everything else — the badges, the self-time bars, the insights — behaves exactly as it does for a query you ran here.

The plan is read on your machine. Pasting a plan into a website means sending table names, index names, filter conditions and sometimes literal values from production to a third party; this exists so you don't have to.

Import & export

Import CSV with header mapping and a type/error preview before committing. Export to CSV, JSON, or Markdown. CSV export is spreadsheet-safe by default (formula injection neutralized); a toggle enables raw output.

Read-only mode

Turn on read-only for a connection and TupleNest issues a PostgreSQL read-only session characteristic — the server refuses writes, not just the UI. It's an accident guardrail; for strict enforcement, use a database role without write privileges.

Dangerous statements

Statements like DELETE/UPDATE without a WHERE, or DROP/TRUNCATE, trigger a confirmation. The check masks comments and strings first, so a commented-out WHERE can't disarm it.

Security & privacy

No telemetry, no account, no cloud. The only network destinations are your PostgreSQL host, your SSH host (if configured), and the GitHub update endpoint. The full model is on the Security page.

Settings

Configure appearance, editor and query preferences, history retention, export defaults, update behaviour, and keyboard shortcuts. Query-history retention and the production audit log are both under your control.

Troubleshooting

  • Cannot connect — run Test to see which stage fails (DNS, TCP, or auth).
  • TLS certificate failure — check the mode and, for a private CA, set the custom CA path.
  • SSH host-key failure — the key changed; verify with your admin before trusting it.
  • Keychain access error — grant TupleNest keychain permission when the OS prompts.
  • Query timeout / large result — adjust the statement timeout or narrow the query; the grid caps are configurable.
  • Update failure — betas update by manual download; the auto-updater targets stable releases.
  • Logs — rotating log files live in the app-data directory (owner-only on Unix).
