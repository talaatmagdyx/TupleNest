The core idea

Safety is the product.

Every guarantee below names where it is enforced — by the server where possible, by reviewed SQL where not — because "we're careful" is not an architecture. Every row has a test that performs the misuse and asserts the refusal.

GuaranteeHow it's actually enforced
🔒 Read-only profilesPostgreSQL itself: SET SESSION CHARACTERISTICS … READ ONLY. The server refuses writes — not a client-side promise.
⚠️ Destructive guardUPDATE/DELETE without WHERE, plus DROP / TRUNCATE / ALTER / GRANT…, ask before running on prod and staging. SQL is masked first — comments, strings, dollar-quotes — so a commented-out WHERE can't disarm it. Thirteen adversarial bypasses are pinned by tests. Best-effort by design: the seatbelt light, not the seatbelt.
✏️ Safe row editingOnly single-table SELECTs carrying the full primary key are editable; joins, groups, CTEs and derived tables are refused with a reason. You review the exact generated UPDATEs before anything runs. Writes are parameter-bound, keyed by the full PK, and guarded with IS NOT DISTINCT FROM on old values — an edit racing another session hits 0 rows and is refused, not silently won.
🔁 Transactions own their tabOne session serves all tabs, so the tab that opened a transaction owns it. Commit from another tab is refused and names the owner. Closing the window mid-transaction always prompts.
🔑 CredentialsOS keychain only — Keychain / Credential Manager / Secret Service. Never in config files, state, logs, or IPC payloads; the backend resolves opaque references.
🛡 TLS fails closedverify-full by default; a bad chain, expired cert or wrong name is an error, never a silent downgrade. verify-ca is real and distinct — chain checked, name knowingly skipped — for SSH tunnels where you dial localhost.
🧷 SSH tunnels pin host keysFingerprint or known_hosts; an unknown or wrong host key fails closed. Tested against a live sshd in CI.
📦 Signed auto-updatesVerified against a minisign public key compiled into the binary. The release host serves bytes; it never holds the private key. Updating is refused while a transaction is open.
Product tour

A tour of the real thing.

Unretouched captures of the shipping build against a local PostgreSQL 18.

01

Connections that prove themselves

A staged probe — DNS, TCP, auth, server version — tells you which layer failed instead of "could not connect". Environments (dev / test / staging / prod) change the window's ambient color and arm the production guards.

The password field says where the password goes, because a promise you can't see is a promise you can't check.

02

Autocomplete that read your schema

Tables after FROM, columns of the in-scope tables in WHERE, alias. resolving to that table's columns, schema. to its tables. Comments and string literals are masked before context is computed, so nothing inside them ever poisons a suggestion.

Plus: Format SQL, EXPLAIN / EXPLAIN ANALYZE plans, $1..$n parameter binding, searchable history, snippets in the command palette.

03

A grid that streams and tells the truth

Virtualized and backpressured: rows arrive in bounded batches, the first 100,000 are kept under both a row cap and a byte budget, and the footer reports truncation honestly. A checksum test pins that nothing is lost or duplicated in transit.

Unknown types are decoded from what the type is, never guessed from what the bytes look like — money renders as money or as visibly-raw hex, never as a plausible wrong number. Full keyboard navigation, real role="grid" semantics.

04

The schema, as a picture

The ER view draws the foreign-key graph and names every constraint. Alongside it: schema diff (columns, types, nullability, PKs), find-usages and rename across tabs, plan-vs-plan comparison with a "new sequential scan" regression flag, and partition tree browsing.

05

Operate, not just query

Live backends, cache hit ratio, commits and rollbacks, blocked locks, and per-session detail — with cancel and terminate. Terminate always confirms; on production it makes you type the pid.

Health panels for index usage, vacuum & bloat, and pg_stat_statements top queries round it out.

📥 CSV import wizard

RFC-4180 parsing — quoted delimiters, escaped quotes, embedded newlines, BOM. You review inferred names and types before anything runs; inserts are batched inside one transaction.

📜 Production audit log

Everything executed on a prod-tagged connection is kept, full SQL, in a dedicated audit log — while regular history omits prod SQL for privacy. Accountability and privacy, separately.

🔎 Global object search

One box over every schema, table, view, column, index and function the cache knows, with instant results served locally.

⚡ Metadata cache

Explorer and completion answer from a local SQLite cache that refreshes live and serves offline — a schema with 13,000 relations opens as a tree, not a spinner.

🔐 Crash logs stay local

No telemetry, no account, no cloud. Panic reports are written to local disk with key/value and URL-credential redaction, and are never uploaded.

♿ Accessible by construction

Real dialog semantics with focus traps on all 28 modals, roving-tabindex grid keyboard model, prefers-reduced-motion respected.

Open the DMG, then drag TupleNest.app onto the Applications shortcut inside it — the drag is the install. Launch from Applications: right-click → Open → Open. Double-clicking offers no way through.

🐧 Linux
glibc 2.35+ · webkit2gtk-4.1
Get AppImage / deb / rpm →
AppImage, dpkg lock, keyring

chmod +x TupleNest_*.AppImage && ./TupleNest_*.AppImage — no root, no package manager.

"Could not get lock /var/lib/dpkg/lock-frontend" — that's unattended-upgrades holding the lock legitimately. Never delete the lock file; use the AppImage or wait a few minutes.

Saved passwords need a keyring daemon (GNOME Keyring / KWallet). Connecting works without one; saving doesn't.

Gatekeeper is a funding problem: an Apple account and notarization make it stop. SmartScreen is not the same — a certificate puts a real name on the installer, but the warning stays until each release earns reputation. Neither dialog tells you what you actually downloaded. The checks below do.

Trust, but verify

Verify what you downloaded.

Two checks, honestly labeled — because they prove different things.

# catches a corrupted or truncated download
shasum -a 256 -c SHA256SUMS --ignore-missing

# binds the file to the CI build that produced it — signed by GitHub's
# OIDC identity, not by anything sitting in the release. The one worth running.
gh attestation verify TupleNest_<file> --repo talaatmagdyx/TupleNest

SHA256SUMS sits beside the installers and is unsigned — whoever could swap an installer could swap it too. The attestation can't be swapped. CycloneDX SBOMs (one per Rust crate, one for the npm tree) ship with every release. Auto-updates verify a minisign signature against a key compiled into the binary and need nothing from you.

Fair questions

Asked, answered honestly.

Including the ones a marketing page would skip.

Why another PostgreSQL client?

Because every one we used would happily run DELETE FROM orders with no WHERE against production. Mature tools exist — DBeaver, pgAdmin, TablePlus — and if you need multi-engine support today, use them. TupleNest exists for one reason: a safety model that names its enforcement, on a fast native app, MIT-licensed.

Is my data safe? Where does anything go?

Nothing you do in TupleNest leaves your computer. No account, no analytics endpoint, no server run by this project. The app talks to two things: the databases you point it at, and — once, at startup — GitHub, to ask whether a newer release exists. Passwords live in your OS keychain. Row data is held in memory only while a result is on screen. PRIVACY.md lists every file it writes, so you can go look.

Why is it unsigned? Should that worry me?

Signing costs money this project doesn't have yet — and on Windows, a standard certificate wouldn't even remove the warning (reputation is per-file and resets each release; only EV skips it). What you can do instead is stronger than either dialog: gh attestation verify cryptographically binds your download to this repository's CI build at a specific commit. It's in the verify section.

What's missing? What should I not expect yet?

One engine: PostgreSQL 13+, done properly, before anything else is attempted. The CSV importer reads the whole file into memory — fine for real-world imports, not the 10 GB one. The destructive-statement guard is best-effort by design (the seatbelt light; read-only profiles are the seatbelt — the server enforces those). And on Linux, saved passwords need a keyring daemon.

How is a beta this heavily tested — and what broke anyway?

1,600+ frontend tests including a suite that executes each documented misuse and asserts the refusal; contract tests against live PostgreSQL 13/15/17; SSH tunnel tests against a real sshd. And still: beta.1 shipped with a close button that did nothing — one missing ACL permission, its rejection swallowed inside a library callback, invisible to every test. beta.2 fixed it and added a test for that whole class of gap. The commit history keeps the story, including the wrong fix that came first, because being checkable is the product.

Is it really free? What's the catch?

MIT-licensed, source on GitHub, built in public CI. There is no paid tier, no account, and no data collection to monetize. The only "catch" is the honest one: it's early software, and what it needs from you is feedback — there's an issue template literally called "nothing broke, it just felt wrong."
